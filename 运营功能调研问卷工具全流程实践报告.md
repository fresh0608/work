# 运营功能调研问卷工具全流程实践报告

## 一、这次实践要解决什么问题

这次不是单纯做一份问卷，而是要解决一个更具体的问题：产品经理希望了解多个功能在运营实际使用中的价值、问题和改进方向，同时又希望从反馈里侧面看出功能设计质量。

一开始看起来可以直接用飞书表单做，但很快暴露出几个现实问题：

- 功能很多，运营不可能每个都熟悉。
- 如果每个功能都问同一套问题，反馈会很空。
- 如果问题写成“完整性、严谨性、架构设计能力”这类产品语言，运营很难回答，也容易误导。
- 如果让产品经理手动把每个问题录进飞书表单，成本高，而且后续调整也麻烦。
- 普通表单只能收答案，不能很好地沉淀每个人的填写快照、IP、设备、每个功能的横向统计。

所以最后目标从“做一份问卷”调整为“做一个轻量的调研工具”：运营端负责简单填写，后台负责汇总分析，产品侧再从数据里判断功能设计问题。

## 二、核心思路：不要让运营回答产品经理的问题

这次最重要的判断是：运营不是来评价“产品经理能力”的，运营只需要描述自己的使用感受。

也就是说，运营端的问题应该问：

- 这个功能能不能帮你更快判断客户？
- 哪一步最容易卡住？
- 哪个点你觉得确实有用？
- 如果只能先改一个地方，你希望改哪里？

不应该直接问：

- 功能是否完整？
- 架构是否严谨？
- 设计是否合理？
- 是否体现产品经理业务理解能力？

这些问题可以作为后台分析维度，但不应该出现在问卷里。

### 示例：把产品语言改成运营语言

原始问题：

> 购买、合同、财务、履约、线索交付链路是否完整？

问题在于，“完整”是产品评审语言，而且购买、合同、财务、履约规则很多本来就来自业务和运营约定。产品在这里更应该被评价的是：系统有没有把这些信息承接好、展示清楚、减少人工核对。

改法：

> 页面里的套餐信息、购买对象和权益内容，能不能让运营快速核对客户买的是哪一类套餐？

这样运营评价的是“系统信息是否好核对”，而不是评价业务规则是否合理。

再比如：

原始问题：

> 方案层次、分配、退款/履约边界是否讲清楚？

改法：

> 运营在处理购买后开通、履约和线索交付时，系统能不能减少反复查信息、问人确认或手工核对？

以及：

> 套餐生效、线索发放、交付异常这些状态变化，系统里有没有足够的记录方便追踪和对账？

这个改法把“规则是否设计得好”拆成了运营能感知的系统实现问题：信息是否集中、状态是否可追踪、对账是否方便。

## 三、问卷结构怎么设计

最终问卷没有采用“一页铺开所有题”的方式，而是设计成三步。

### 第一步：先选择熟悉的功能

不是每个运营都用过所有功能，所以不能强制所有人评价全部功能。

设计原则是：

- 熟悉几个就选几个。
- 可以只评价一个。
- 也鼓励多评价几个，但不强制。
- 不设置上限，避免人为限制有效反馈。

这样既降低填写压力，也避免不熟悉的人硬答，污染数据。

### 第二步：逐个评价已选功能

选择功能后，进入详细评价。每次只处理一个功能、一个步骤。

每个功能包含四类问题：

1. 使用感受  
   不是打分题，而是用“很难用、不太顺、一般、比较顺、很好用”表达感受。

2. 最大问题  
   让运营选出最影响使用的一类问题，并写具体场景。

3. 喜欢的点  
   提供快捷选项，也允许自定义。选完后必须说明为什么觉得有价值。

4. 不喜欢/想改的点  
   同样支持快捷选项和自定义，并要求写出具体卡点或希望怎么改。

这里有一个关键约束：已选功能的必填项必须全部完成后，才能进入总评和提交。

原因很简单：如果总评和提交一直可见，用户容易跳过功能细节，最后只写几句泛泛而谈的话。调研真正有价值的部分，往往在“具体功能、具体步骤、具体场景”里。

### 第三步：整体补充说明

只有当已选功能都填写完整后，才出现整体补充说明。

整体问题不再问复杂概念，只保留三类：

- 最顺手的地方
- 最不顺手的地方
- 最希望先改的一件事

这三个问题用于捕捉横向感受，而不是替代每个功能的详细评价。

## 四、喜欢和不喜欢为什么要重点设计

这次问卷里，“喜欢的点”和“不喜欢/想改的点”是重点，不是简单让用户填一段描述。

原因是：

- 只让用户写开放题，很多人会不知道怎么写。
- 只给固定选项，又会限制真实表达。
- 只选选项但不写原因，后续很难判断价值来自哪里。

所以最后采用了“快捷选项 + 自定义 + 场景说明”的组合。

### 示例：喜欢的点

对于“朋友圈私域运营功能”，快捷选项可以是：

- 互动集中处理
- 私聊触达方便
- 标记已处理清楚
- 跟进优先级更好判断

但用户也可以自己写一个点。无论选快捷项还是自定义，都要补充一个场景：

> 什么时候让你觉得它好用？

这样后台看到的不只是“互动集中处理被选了 6 次”，还能看到它为什么有价值。

### 示例：不喜欢/想改的点

对于“线索自动分配”，快捷选项可以是：

- 分配原因不透明
- 公平性不放心
- 未命中兜底不清楚
- 人工干预不方便

同时要求补充：

> 哪一步让你犹豫、容易选错，或希望怎么改？

这类描述比单纯的“不好用”更能指导后续改版。

## 五、从问卷到工具：为什么做简单前后端

如果继续用飞书表单，需要手工录入大量题目，而且每次调整都要重新编辑表单。功能多、问题多、逻辑多时，飞书表单会变得很难维护。

所以这次改成了一个轻量前后端工具：

- 前端负责问卷填写体验。
- 后端负责保存提交数据。
- 后台负责查看汇总结果、每个人的填写快照、IP、设备信息。
- 配置和问题可以在代码里统一维护。

这个选择不是为了“做系统”，而是因为这个问卷已经超出了普通表单的适用范围。

### 工具里保留的几个关键能力

1. 运营角色手工填写  
   不预设固定角色，避免选项不准。

2. 不问联系方式和使用频率  
   减少无关负担，聚焦功能反馈。

3. 管理后台加密码  
   问卷页不出现后台入口，后台需要密码访问。

4. 记录提交快照  
   后续能看每个人原始填写内容，而不是只看汇总。

5. 记录 IP 和设备信息  
   用于判断是否有重复提交或样本异常。

## 六、UI 和交互迭代过程

这次 UI 不是一次就定下来的，中间经历了多轮修正。

最早的问题是：页面把大量问题、选项、说明直接堆在一起，视觉上很压迫，运营会不想认真填。

后面逐步做了几个调整：

- 从“一页铺开”改成“先选功能，再进入详细评价”。
- 从“大块表单”改成“左侧已选功能清单，右侧填写当前功能”。
- 每个功能显示完成状态，明确还差哪些必填项。
- 总评和提交按钮只在所有已选功能完成后出现。
- 去掉页面里的内部评价话术，例如“产品设计能力”“设计靠谱”“不做个人考核”等。
- 后台密码不再直接显示在页面上。

### 一个典型 UI 问题

曾经页面里出现过这样的结构：

- 顶部是已选功能标签。
- 中间是完成提示。
- 下方又是当前功能标题。
- 再下面是还差几项。
- 然后才是当前步骤。

看起来信息很多，但没有主次。用户不知道先看哪里。

后来改成：

- 左边：已选功能列表和完成状态。
- 右边：当前功能名称、说明、当前步骤。
- 当前功能里明确展示“还差哪些项”。

这样信息仍然完整，但用户的注意力更集中。

## 七、部署过程踩过的坑

部署部分不展开太多，但有几个坑值得记录。

### 坑 1：一开始误以为是 SSH Key 问题

部署过程中曾经尝试通过 SSH key 推送 GitHub，但 GitHub 没接受 key。后来复盘发现，这不是第一次成功路径，也不是最该优先排查的方向。

真正可用的路径是 Git Credential Manager，也就是 Git 在 Windows 上弹出的网页登录授权。第一次成功推送靠的是这个机制，而不是手工配置 SSH key。

经验是：如果之前某条链路成功过，不要凭感觉换路线，应该先复盘成功链路。

### 坑 2：Render 服务名未必已经存在

访问 `operator-survey-app.onrender.com` 返回 404 时，一开始容易误以为是代码部署失败。

实际返回头里有 `x-render-routing: no-server`，意思是 Render 还没有对应服务，而不是服务启动失败。

判断方式：

- 如果是 `no-server`，说明服务还没创建或服务名不对。
- 如果服务存在但构建失败，应该去 Render 的 Logs 看构建日志。
- 如果服务启动但页面错误，再看应用日志。

### 坑 3：免费部署会慢

Render 免费服务创建和首次启动会比较慢。截图里同步任务显示 Running 一分钟以上是正常的。

最终看到两个绿色对勾，表示：

- Blueprint 同步成功。
- Web Service 创建成功。

之后再访问问卷地址和后台地址做实际验证。

## 八、最终成果

最终产出包括：

- 一个运营填写问卷页面。
- 一个带密码的后台页面。
- 一套按功能配置的问题。
- 一套喜欢/不喜欢点的快捷选项和自定义机制。
- 一个可以部署到 Render 的 Node 应用。

线上地址：

- 问卷页：`https://operator-survey-app.onrender.com/`
- 后台页：`https://operator-survey-app.onrender.com/admin`

后台密码使用 Render 环境变量 `ADMIN_PASSWORD` 设置。

## 九、这次实践的关键经验

### 1. 调研问题要从回答者视角出发

运营能回答的是“我在哪里卡住”“我觉得哪里有用”“我希望先改哪里”。

产品侧真正关心的“业务理解、产品设计、架构严谨性、体验能力、AI 知识深度”，应该放到后台分析维度里，而不是直接暴露给运营。

### 2. 不熟悉的功能不要强制评价

强制所有人评价所有功能，看似样本更完整，实际会引入大量低质量答案。

更好的方式是先让用户选择熟悉功能，再鼓励多选，但不强制。

### 3. 开放题不能裸奔

纯开放题会让用户不知道怎么写。

更好的方式是：

- 给几个快捷选项降低填写门槛。
- 保留自定义选项避免限制表达。
- 要求补充具体场景，保证反馈可用。

### 4. 工具体验会影响反馈质量

如果页面压迫、层级混乱、看不到进度，用户会快速敷衍。

问卷工具本身也需要产品设计：

- 当前在填什么？
- 还差什么？
- 下一步是什么？
- 为什么现在不能提交？

这些问题不解决，问卷再专业也很难收集到好数据。

## 十、后续可以怎么改进

### 1. 沉淀问卷设计 Skill

这次很多判断可以沉淀成一个专门的 Skill，用于以后快速生成类似调研工具。

这个 Skill 可以包含：

- 如何把产品评审语言改成一线用户语言。
- 如何设计“熟悉功能选择”机制。
- 如何设计喜欢/不喜欢点。
- 如何把后台分析维度隐藏在问卷问题背后。
- 如何判断哪些问题不该问运营。

例如可以沉淀一条规则：

> 如果问题里出现“完整性、严谨性、架构、产品设计能力、业务理解程度”等词，默认不直接给运营看，需要改写成具体工作场景。

### 2. 沉淀部署 SOP

这次部署踩坑说明，后续需要固定流程。

建议 SOP：

1. 本地测试通过。
2. 提交代码。
3. 先确认上次成功推送方式。
4. 优先复用 Git Credential Manager。
5. 推送到 GitHub main。
6. Render Blueprint 同步。
7. 等服务创建完成。
8. 验证问卷页、后台页、接口。

不要一上来就切换到 SSH key、Vercel 或其他平台，除非确认原链路不可用。

### 3. 后台数据存储升级

当前数据保存在服务文件里，适合短期收集和小范围调研。

如果后续要长期使用，建议接数据库，例如：

- Supabase
- Render Postgres
- 飞书多维表格 API

这样可以避免免费服务重启、文件丢失或多人同时写入的问题。

### 4. 形成“问题库”

以后每类功能可以沉淀一组高质量问题，而不是每次从零开始写。

例如：

AI 功能类问题可以沉淀：

- AI 给出的结果是否能直接支持运营判断？
- AI 不确定的地方是否有提示？
- 结果依据是否能看懂？

线索分配类问题可以沉淀：

- 分配依据是否清楚？
- 异常情况是否有人工处理入口？
- 信息优先级是否能支持快速判断？

商业化链路类问题可以沉淀：

- 套餐信息、购买对象和权益内容是否方便运营快速核对？
- 购买后开通、履约和交付信息是否减少反复查找、问人确认和手工核对？
- 生效、发放、异常等状态变化是否有记录，方便追踪和对账？

## 十一、以后类似项目可以怎么做

可以按下面这套流程复用。

### 第一步：先判断是不是普通表单能解决

如果满足以下条件，飞书表单就够用：

- 问题少。
- 没有复杂分支。
- 不需要后台统计。
- 不需要记录提交快照。

如果出现以下情况，就适合做轻量工具：

- 功能多。
- 不同功能问题不同。
- 需要先选择熟悉功能。
- 需要后台看维度统计。
- 需要记录每个人原始答案和来源信息。

### 第二步：先定义分析目标，再反推问题

不要一开始就写题。

先明确产品侧要看什么：

- 哪些功能有价值？
- 价值点是什么？
- 哪些功能不好用？
- 不好用在哪里？
- 哪些问题反映业务理解不足？
- 哪些问题反映规则设计不清？
- 哪些问题反映 AI 能力边界没处理好？

然后再把这些目标翻译成用户能回答的问题。

### 第三步：把用户填写成本控制住

控制成本不是简单砍题，而是设计好填写路径。

这次的做法是：

- 不强制评价全部功能。
- 每次只显示一个功能。
- 每个功能只问最关键的几类问题。
- 用快捷选项降低输入成本。
- 用具体场景保证反馈质量。

### 第四步：后台负责“二次解释”

问卷前台不做能力评价，后台才做归因分析。

例如运营回答：

> 分配结果看得到，但不知道为什么分给这个人。

后台可以归因到：

- 规则解释不足。
- 架构与分配逻辑不透明。
- 运营缺少异常判断依据。

这才是产品经理需要看的部分。

## 十二、可以继续沉淀的资产

这次实践结束后，不应该只留下一个问卷工具，至少可以沉淀四类资产：

1. 问卷设计方法  
   如何从产品目标反推运营问题。

2. 功能问题库  
   AI 类、线索类、商业化类、投放类、触达类分别有哪些可复用问题。

3. 轻量调研工具模板  
   前端问卷、后台统计、部署配置可以复用。

4. 部署检查清单  
   本地测试、Git 推送、Render 构建、线上验证形成固定 SOP。

真正有价值的不是这一次做出了一个页面，而是以后遇到类似“多功能、多角色、要看真实使用反馈”的调研场景时，可以更快、更稳地做出一套能用的工具。
